validate an in-\/wasm-\/memory array 
\begin{DoxyTemplParams}{Template Parameters}
{\em T} & \\
\hline
\end{DoxyTemplParams}
When a pointer will be invalid we want to stop execution right here right now. This is accomplished by forcing a read from an address that must always be bad. \doxylink{struct_a}{A} better approach would probably be to call in to a function that notes the invalid parameter and host function and then bubbles up a more useful error message; maybe some day. Prior to WASM\+\_\+\+LIMITS the code just simply did a load from address 33MB via an immediate. 33MB was always invalid since 33MB was the most \doxylink{namespace_w_a_s_m}{WASM} memory you could have. Post WASM\+\_\+\+LIMITS you theoretically could have up to 4GB, but we can\textquotesingle{}t do a load from a 4GB immediate since immediates are limited to signed 32bit ranges.

So instead access the first\+\_\+invalid\+\_\+memory\+\_\+address which by its name will always be invalid. Or will it? No... it won\textquotesingle{}t, since it\textquotesingle{}s initialized to -\/1\texorpdfstring{$\ast$}{*}64\+KB in the case \doxylink{namespace_w_a_s_m}{WASM} has {\itshape no} memory! We actually cannot clamp first\+\_\+invalid\+\_\+memory\+\_\+address to 0 during initialization in such a case since there is some historical funny business going on when end==0 (note how jle will {\itshape pass} when end==0 \& first\+\_\+invalid\+\_\+memory\+\_\+address==0)

So instead just bump first\+\_\+invalid\+\_\+memory\+\_\+address another 64KB before accessing it. If it\textquotesingle{}s -\/64KB it\textquotesingle{}ll go to 0 which fails correctly in that case. If it\textquotesingle{}s 4GB it\textquotesingle{}ll go to 4GB+64\+KB which still fails too (there is an entire 8GB range of \doxylink{namespace_w_a_s_m}{WASM} memory set aside). There are other more straightforward ways of accomplishing this, but at least this approach has zero overhead (e.\+g. no additional register usage, etc) in the nominal case.
